# 框架产品化待处理清单

本文档记录当前项目从“内部平台开发底座”进一步升级为“通用后台 Web 框架”的待处理事项。当前框架已经可以支撑 Metrix 后续业务模块开发，但距离让其他开发者开箱即用，还需要补齐文档、模板、模块化注册、迁移策略、模块接入机制和测试覆盖。

## 1. README、快速启动、初始化与部署说明

**现状**

- 项目已有基础代码结构和开发文档，但 README、快速启动、初始化、部署说明还不够完整。
- 新开发者需要阅读多份文档和代码后才能完整理解项目启动、初始化、配置和开发流程。

**待处理**

- 完善根目录 README，覆盖项目定位、技术栈、目录结构、环境要求、安装依赖、启动前端、启动后端、初始化系统和默认管理员说明。
- 补充开发环境与生产环境的配置说明，包括环境变量、运行时目录、SQLite/MySQL 切换方式、离线资源约束和日志目录。
- 补充部署说明，包括后端运行方式、前端构建产物、反向代理建议、静态资源部署、备份恢复和常见问题。
- 给出最短可运行路径，让新开发者可以按文档从空环境启动到登录系统。

**完成标准**

- 新开发者只阅读 README 和必要链接即可完成本地启动、初始化、登录和进入开发。
- 部署人员可以根据文档完成内网环境部署，不需要反复查代码确认配置项。

## 2. 标准示例模块

**现状**

- 当前已有用户、角色权限、公告、操作日志、Token、系统设置等真实功能。
- 这些功能可作为参考，但对新业务开发者来说仍然偏复杂，缺少一个最小完整示例。

**待处理**

- 新增一个标准示例模块，例如 `demo-crud`。
- 示例模块需要覆盖后端 API、数据库模型、schema、repository、service、权限校验、审计日志和测试。
- 示例模块需要覆盖前端 API 封装、页面、表格分页、筛选、操作按钮、权限控制、i18n、路由和菜单。
- 示例模块应保持轻量，重点展示标准开发路径，不引入真实业务复杂度。

**完成标准**

- 开发者可以复制或参考示例模块，快速完成一个新的业务 CRUD 页面。
- 示例模块能够体现当前项目的权限、审计、i18n、分页、表格滚动和本人/他人数据权限预留规则。

## 3. 权限、路由、菜单与 i18n 模块化注册

**现状**

- 前端页面、路由和菜单已经集中在 `web/src/router/page-registry.ts`。
- 后端权限种子已经集中在 `server/app/core/permissions.py`。
- 前端语言资源已经按语言 JSON 拆分，并支持动态加载。
- 新增功能时仍需要开发者按规范分别维护前端路由菜单、后端权限种子和语言资源。

**待处理**

- 设计轻量模块声明机制，让每个业务模块可以声明自己的页面、菜单、权限、功能开关和语言资源。
- 前端优先考虑基于 `web/src/modules/*/index.ts` 的模块 manifest，由框架层统一收集并生成路由、菜单、fallback 和标题。
- i18n 优先考虑支持模块级语言包，例如 `web/src/modules/<module>/i18n/<locale>.json`，加载语言时自动合并基础语言包和模块语言包。
- 后端优先考虑显式模块列表，由框架层统一注册 API router、收集权限种子并同步权限。
- 保持权限 code 稳定，继续使用 `route:<page>` 和 `action:<resource>:<action>` 规则。

**完成标准**

- 新增一个业务模块时，开发者主要在模块目录内完成声明和实现，不需要到多个框架核心文件中分散注册。
- 权限页面、侧边菜单、路由守卫和语言切换可以从模块声明自动获得所需信息。
- 保留显式模块启用清单，避免过度自动扫描导致排查困难。

## 4. 数据库迁移策略

**现状**

- 当前处于开发期，数据库结构偏向启动时自动同步，适合快速迭代。
- 如果作为通用框架或正式部署底座，需要更明确的迁移策略。

**待处理**

- 明确开发环境和生产环境的数据库迁移边界。
- 评估是否引入正式迁移工具，或保留轻量自维护迁移机制。
- 设计 SQLite 与 MySQL 之间切换时的结构同步、初始化数据、权限种子和系统设置迁移规则。
- 补充版本升级、失败回滚、备份恢复和迁移日志说明。

**完成标准**

- 开发期可以继续快速同步结构。
- 正式部署时数据库变更可追踪、可回放、可回滚，并且不依赖开发者手工改库。

## 5. Submodule 与业务模块接入机制

**现状**

- 项目已有多仓库与 submodule 的设计方向。
- 当前初版 Web/server 框架还没有形成真正的业务模块接入机制。

**待处理**

- 明确业务模块作为 submodule 接入时的目录位置、命名规则、依赖边界和开发规范。
- 明确前端模块、后端模块、语言资源、权限种子、API router、数据库迁移和测试如何随 submodule 接入主项目。
- 明确模块启用、禁用、升级和删除时的处理方式。
- 保持接入机制轻量，不在当前阶段引入复杂插件系统。

**完成标准**

- 一个业务模块可以作为独立仓库维护，并通过 submodule 接入主项目。
- 接入后只需要少量显式配置即可参与路由、菜单、权限、i18n、API 和测试。

## 6. 前端与后端测试覆盖

**现状**

- 后端已有核心测试，前端主要依赖类型检查、构建和浏览器人工回归。
- 作为通用框架，关键能力仍需要更稳定的自动化回归保障。

**待处理**

- 后端继续补充权限、用户、公告、设置、Token、审计日志和后续业务模块的单元测试与接口测试。
- 前端补充关键工具函数、store、权限按钮、路由守卫、i18n 和核心页面交互测试。
- 建立最小浏览器回归用例，覆盖登录、权限菜单、表格分页筛选、表单校验、语言切换和 404 等关键路径。
- 将测试命令和覆盖范围写入 README 或开发指南。

**完成标准**

- 关键框架能力具备可重复执行的自动化测试。
- 新增业务模块可以按示例补充对应测试，不完全依赖人工浏览器验证。

## 建议优先级

1. 完善 README、快速启动和部署说明。
2. 新增一个标准 `demo-crud` 示例模块。
3. 设计并落地轻量模块 manifest，优先解决路由、菜单、权限和 i18n 的分散注册问题。
4. 明确数据库迁移策略。
5. 明确 submodule 业务模块接入机制。
6. 补齐前后端关键自动化测试。

## 当前建议

短期不要直接引入复杂插件系统。优先把“新增模块需要改哪些地方”压缩成清晰的模块声明和示例模板，让开发者可以少改核心代码、少查上下文、少踩权限和 i18n 的坑。等业务模块数量增加后，再决定是否需要进一步做自动发现、脚手架或插件化加载。
