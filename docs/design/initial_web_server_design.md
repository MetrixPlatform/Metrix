# 初版 Web 与后端详细设计稿

## 1. 目标与范围

初版只实现平台基础框架，先把登录、注册、账号、权限和基础页面跑通。暂不接入 FTP/SFTP、文件处理、业务数据库处理、脚本任务和调度任务。

初版必须包含：

| 功能 | 类型 | 说明 |
| --- | --- | --- |
| 登录 | 页面 + API | 内网账号密码登录 |
| 注册 | 页面 + API | 用户提交注册申请，等待管理员审核 |
| 用户管理 | 页面 + API | 管理员维护用户资料、状态、角色、密码，并审核注册申请 |
| 个人信息管理 | 页面 + API | 用户维护自己的公司、部门、姓名和密码 |
| 退出登录 | 前端动作 + API | 清理本地登录态并通知后端 |
| 首页 | 页面 + API | 登录后的基础工作台 |
| 权限管理 | 页面 + API | 角色、路由权限和功能权限维护 |

初版原则：

- 轻便、简洁、优雅、可维护。
- 后端使用 Python。
- 网站库使用 SQLite。
- 前端只做 Web。
- 页面框架参考 CapaReport 的 Web 后台工具风格。
- 所有运行期 UI、图标、字体、文档和静态资源必须离线可用。
- 代码按需组织，不写冗余实现，不为未来假设提前堆复杂抽象。

## 2. 技术选型

### 2.1 后端

建议：

- Python。
- FastAPI。
- Uvicorn。
- SQLite。
- SQLAlchemy。
- Pydantic。

说明：

- FastAPI 负责 API、鉴权依赖和 OpenAPI 文档。
- SQLite 作为初版网站库，保存用户、角色、权限和审计记录。
- SQLAlchemy 用于轻量 ORM 和后续数据库迁移弹性。
- Pydantic 用于请求和响应结构校验。
- 数据库访问必须经过统一数据访问层，便于后续在 SQLite 和 MySQL 之间切换。

第一版不引入：

- Celery。
- Redis。
- 外部统一认证。
- 多租户。
- 复杂组织架构。
- 复杂任务队列。

### 2.2 前端

建议：

- Vue 3。
- TypeScript。
- Vite。
- Naive UI。
- 本地图标包。

说明：

- 前端负责页面交互、路由守卫、权限菜单和 API 调用。
- 权限判断以后端为准，前端隐藏菜单和按钮只是交互体验。
- 页面框架参考 CapaReport 的 Web 管理界面：左侧菜单、顶部页头、内容区工作卡片、居中登录卡片、明暗主题。
- 组件、样式和工具函数按需拆分，避免单个 Vue 文件过大。
- UI 组件和图标按需导入，避免首屏 chunk 过大。

## 3. 离线资源规则

系统主要在内网运行，服务器可能无法访问外网。所有运行期资源必须本地化。

必须离线可用：

| 资源 | 要求 |
| --- | --- |
| UI 组件 | 通过本地 npm 依赖构建进入前端产物 |
| 图标 | 使用本地图标包或项目内静态文件 |
| 字体 | 优先使用系统字体；如需自带字体，放入项目静态资源 |
| 图片 | 放入项目静态资源或上传资源目录 |
| API 文档 UI | 使用本地打包资源，不使用 CDN |
| 脚本和样式 | 只来自构建产物，不引用外网地址 |

禁止：

- 禁止使用 CDN。
- 禁止使用在线字体。
- 禁止使用远程图标地址。
- 禁止使用外网图片、脚本、样式或 iframe。
- 禁止运行期依赖外网 API 加载 UI 配置或文档资源。
- 禁止把外网静态资源 URL 写死在前端代码、HTML 模板、后端返回内容或配置样例中。

验收：

- 断开服务器外网后，登录、注册、首页、用户管理、个人信息管理、权限管理和 API 文档仍可正常加载。
- 构建产物中不应出现外网静态资源引用。
- 业务数据源地址必须通过明确配置接入，不能作为 UI 资源隐式依赖。

## 4. 目录结构

初版先使用单仓库目录，不拆 submodule。待框架稳定后再按职责拆分。

```text
Metrix/
  web/
    src/
      api/
      components/
      router/
      stores/
      styles/
      views/
  server/
    app/
      api/
      core/
      db/
      models/
      repositories/
      schemas/
      services/

  runtime/
```

目录职责：

| 目录 | 职责 |
| --- | --- |
| `web/src/api` | API 请求封装和类型适配 |
| `web/src/components` | 通用组件，如页面壳、权限按钮、表格工具栏 |
| `web/src/router` | 路由配置和路由守卫 |
| `web/src/stores` | 登录态、当前用户、权限、主题状态 |
| `web/src/styles` | 全局样式和主题变量 |
| `web/src/views` | 页面视图，只保留页面编排和少量页面状态 |
| `server/app/api` | API 路由 |
| `server/app/core` | 配置、鉴权、权限依赖、异常处理 |
| `server/app/db` | 数据库连接、会话、初始化和迁移入口 |
| `server/app/models` | 数据库模型 |
| `server/app/repositories` | 数据访问层，隔离 ORM 查询和数据库差异 |
| `server/app/schemas` | 请求和响应结构 |
| `server/app/services` | 登录、用户、角色、权限、审计等服务 |
| `runtime/` | 本地运行数据目录，不进入版本库 |

## 5. 实现质量规则

本项目初版优先追求轻便、简洁、优雅。实现要解决当前明确需求，不引入不必要的复杂抽象，不为后续可能出现但尚未确认的场景提前堆代码。

通用规则：

- 遵循 KISS 和 YAGNI，优先选择最简单、清晰、可验证的实现。
- 代码职责清晰，API、服务、数据访问、模型和前端页面各司其职。
- 不写重复逻辑，不做平行实现。
- 不添加暂时用不到的框架、依赖、插件或复杂基础设施。
- 不为了“看起来完整”保留空方法、占位函数、未接入常量或未使用配置。
- 不保留临时代码、调试输出、测试假数据、注释掉的旧代码。
- 每次修改、功能或修复完成后，必须检查并清理冗余导入、函数、变量、常量、方法、组件、样式和依赖。
- 错误处理要清晰，但不要为了少数极端场景写复杂分支。
- 文档、接口、命名和目录要保持一致，不让后续实现者猜。

代码完成后的清理检查：

| 检查项 | 要求 |
| --- | --- |
| 导入 | 删除未使用导入，避免宽泛导入 |
| 函数和方法 | 删除未调用或重复职责的函数和方法 |
| 变量和常量 | 删除未使用变量、临时常量和调试值 |
| 组件 | 删除未使用组件，合并重复组件 |
| 样式 | 删除未使用样式，复用公共样式 |
| 日志 | 删除调试日志，只保留必要错误或审计日志 |
| 依赖 | 不保留未使用依赖 |

## 6. 数据库访问层设计

初版网站库使用 SQLite，但数据库访问必须预留 SQLite 和 MySQL 之间切换的能力。这里的“预留”只做必要边界，不做复杂多数据库框架。

设计原则：

- API 层不直接操作数据库模型。
- 服务层不拼接底层 SQL。
- 数据访问集中放在 `server/app/repositories`。
- 数据库连接、会话、事务和初始化集中放在 `server/app/db`。
- `server/app/db` 需要作为数据库中间件和适配入口，统一处理 SQLite、MySQL 的连接配置、engine 创建、session 生命周期和初始化流程。
- 业务代码只依赖统一的 session、repository 和 service，不感知当前使用 SQLite 还是 MySQL。
- 业务服务通过 repository 读写数据。
- SQLAlchemy 模型保持通用字段类型，避免使用只有某个数据库支持的特殊能力。
- 需要分页、排序、筛选时，在 repository 内统一处理。
- 如确实存在数据库方言差异，只能收敛在 `db` 或 repository 内，不允许散落到 API、service 或前端代码中。

数据库切换策略：

| 层级 | 职责 |
| --- | --- |
| 配置层 | 提供数据库类型和连接字符串 |
| `db` 层 | 作为数据库适配中间件，创建 engine、session、初始化表并屏蔽数据库差异 |
| model 层 | 定义通用模型 |
| repository 层 | 封装查询、写入、分页和事务边界 |
| service 层 | 编排业务规则，不关心 SQLite 或 MySQL 差异 |

初版优先实现 SQLite 所需能力，但数据库配置、连接、初始化和会话管理必须按 SQLite/MySQL 可切换的边界设计。不能把 SQLite 文件路径、连接细节和方言差异散落到 API 或服务代码中。

## 7. 前端代码组织规则

前端要避免单文件过大，也要避免每个页面都重复写一套 CSS。

Vue 组织规则：

- 页面组件放在 `web/src/views`，只负责页面编排、状态和调用 API。
- 通用组件放在 `web/src/components`，如布局壳、权限按钮、用户表单、角色表单、权限矩阵、状态标签。
- API 请求封装放在 `web/src/api`。
- 登录态、当前用户、权限和主题状态放在 `web/src/stores`。
- 路由定义和路由守卫放在 `web/src/router`。
- 页面组件过大时，按表格、表单、弹窗、权限矩阵等职责拆成子组件。

CSS 规则：

- CSS 与 Vue 结构分离：可复用样式统一放在 `web/src/styles`，Vue 文件只保留模板、逻辑和少量真正私有的组件样式。
- 全局主题变量、布局、通用卡片、表格工具栏、状态标签、表单间距等放在 `web/src/styles`。
- Vue 文件中不重复写可复用样式，不把页面级公共样式堆在单个组件里。
- 组件私有样式只保留真正只属于该组件的少量样式；一旦多个页面或组件复用，应移动到公共样式或通用组件。
- 相同视觉结构必须复用同一组 class 或通用组件。
- 不为了一个页面临时复制一大段 CSS。

按需导入规则：

- UI 组件按需导入。
- 图标按需导入。
- 路由页面使用懒加载。
- 大型编辑器、文档 UI 或后续重型组件必须独立 chunk。
- 禁止整包导入 UI 库、图标库或工具库；确需使用多个能力时，也要按实际使用项导入。
- 定期关注构建输出，发现主入口 chunk 过大时优先检查全量组件库导入、全量图标导入、同步加载页面和重复依赖。

## 8. Python 代码组织规则

Python 代码要保持短小、直接、可读。文件过长时按职责拆分。

后端分层：

| 层级 | 职责 |
| --- | --- |
| `api` | 路由、请求校验、响应返回 |
| `schemas` | 请求和响应结构 |
| `services` | 业务规则和流程编排 |
| `repositories` | 数据访问 |
| `models` | ORM 模型 |
| `core` | 配置、鉴权、权限校验、异常 |
| `db` | 数据库连接、会话、初始化 |

拆分规则：

- API 文件按资源拆分，如 `auth`、`users`、`roles`。
- Service 文件按业务拆分，如登录、用户、角色、权限、审计。
- Repository 文件按聚合根拆分，如用户、角色、权限、审计。
- 工具函数只放通用能力，不放业务流程。
- 文件明显过长、逻辑过密或职责混杂时，先按资源、流程或职责拆分为小文件，不抽象成复杂基类。
- 单个 Python 文件不承担多个资源的完整业务流程；确需共享逻辑时优先提取小函数或小服务。
- 不使用通配符导入。
- 不保留未使用方法和临时调试代码。

## 9. 实现收尾要求

每个功能完成后都要做收尾检查。

必须检查：

- 是否存在未使用导入。
- 是否存在未使用函数、方法、变量、常量。
- 是否存在重复组件、重复样式、重复 API 封装。
- 是否存在临时日志、调试输出、假数据、临时账号密码。
- 是否存在可复用 CSS 被重复写在多个 Vue 文件中。
- 是否存在本应抽离的公共组件、公共样式或 API 封装仍散落在页面文件中。
- 是否存在全量导入导致大 chunk。
- 是否存在数据库访问绕过 repository。
- 是否存在 SQLite 或 MySQL 连接细节散落到 `db` 层之外。
- 是否存在过长 Python 文件或过大的 Vue 文件。
- 是否存在外网资源引用。

完成标准：

- 代码能运行。
- 权限判断以后端为准。
- 页面资源离线可用。
- 样式和组件复用清晰。
- 数据库访问边界清楚。
- 没有明显冗余代码。

## 10. 页面框架设计

页面风格参考 `E:\Work\code\CapaReport\frontend\src` 中的 Web 后台工具框架，但不引入 Tauri，也不复制业务页面。

### 10.1 整体布局

登录后使用后台管理布局：

- 左侧可折叠菜单。
- 顶部页头。
- 右侧主内容区。
- 顶部右侧放主题切换、当前用户和退出登录。
- 菜单根据路由权限动态过滤。
- 内容区使用工作卡片、表格、表单和弹窗。

布局约定：

| 区域 | 设计 |
| --- | --- |
| 左侧菜单 | 默认宽度约 220px，折叠后约 64px |
| 品牌区 | 显示 Metrix 标识和名称 |
| 菜单项 | 图标 + 文本，选中项使用主色背景和左侧强调条 |
| 顶部页头 | 显示当前页面标题、辅助说明和页面动作 |
| 内容区 | 全高滚动，背景为浅灰或暗色页面背景 |
| 卡片 | 用于表单、列表、详情和权限矩阵 |

### 10.2 视觉风格

沿用安静、工具型后台风格：

- 主色使用稳定蓝色。
- 背景使用浅灰页面背景和白色容器。
- 暗色主题使用深灰页面背景和深色容器。
- 成功、警告、错误分别使用绿色、橙色、红色。
- 圆角保持克制，工具卡片不做夸张装饰。
- 表格、筛选、按钮和弹窗优先保证密度和可读性。

字体：

- 默认使用系统字体。
- 等宽内容使用系统等宽字体。
- 不使用在线字体。

图标：

- 使用本地图标包。
- 按钮和菜单优先使用图标表达含义。
- 图标资源随前端依赖打包。

## 11. 页面清单

### 11.1 登录页

路由：`/login`

访问：未登录用户可访问。

页面内容：

- Metrix 品牌标识。
- 账号输入。
- 密码输入。
- 登录按钮。
- 注册入口。
- 忘记密码提示：联系管理员重置密码。

行为：

- 登录成功后进入首页。
- 登录失败返回统一错误，不提示账号是否存在。
- 已登录用户访问登录页时跳转首页。

### 11.2 注册页

路由：`/register`

访问：未登录用户可访问。

注册字段：

- 账号。
- 密码。
- 确认密码。
- 公司。
- 部门。
- 姓名。

行为：

- 注册后创建待审核用户。
- 待审核用户不能登录。
- 注册成功后提示等待管理员审核。
- 不发送邮件、短信或外部通知。

### 11.3 首页

路由：`/`

权限：`route:dashboard`

页面内容：

- 当前登录用户信息摘要。
- 系统运行状态。
- 当前用户可访问页面概览。
- 待审核用户数量，仅对有用户操作权限的管理员显示。

初版首页只做框架工作台，不放业务统计。

### 11.4 用户管理

路由：`/users`

路由权限：`route:users`

默认查询权限：`action:user:read`

页面内容：

- 用户列表。
- 搜索账号、姓名、公司、部门。
- 按启用状态、审核状态筛选。
- 创建用户。
- 编辑用户资料。
- 启用或禁用用户。
- 删除用户。
- 重置密码。
- 分配角色。
- 审核通过待审核用户。
- 驳回待审核用户。
- 审核通过时分配角色。

字段展示：

- 账号。
- 姓名。
- 公司。
- 部门。
- 审核状态。
- 启用状态。
- 角色。
- 最近登录时间。
- 创建时间。

功能权限：

| 权限 | 页面能力 |
| --- | --- |
| `action:user:create` | 创建用户 |
| `action:user:update` | 修改用户资料 |
| `action:user:delete` | 删除用户 |
| `action:user:operate` | 审核通过、驳回、启用、禁用、重置密码、分配角色 |

约束：

- 不允许删除或禁用最后一个管理员。
- 不允许普通管理员把自己降权到失去权限管理能力。
- 重置密码由管理员设置临时密码或生成一次性临时密码，具体方式实现时再定。
- 只有待审核用户可以审核通过或驳回。
- 审核通过后用户才能登录。
- 驳回后保留记录，用于审计和排查。
- 不分配角色时使用默认普通用户角色。

### 11.5 个人信息管理

路由：`/profile`

访问：登录用户均可访问，不单独配置路由权限。

页面内容：

- 账号，只读。
- 姓名。
- 公司。
- 部门。
- 修改密码。

行为：

- 用户可以修改自己的姓名、公司和部门。
- 修改密码必须提供旧密码。
- 忘记密码不提供自助找回入口。
- 密码修改成功后可继续保持登录，也可以实现为重新登录，后续实现时选择一种并保持一致。

### 11.6 权限管理

路由：`/permissions`

路由权限：`route:permissions`

默认查询权限：`action:role:read`

页面内容：

- 角色列表。
- 创建角色。
- 编辑角色名称和说明。
- 删除角色。
- 权限分配。

权限分配界面：

- 左侧为角色列表。
- 右侧为权限矩阵。
- 权限矩阵按页面分组。
- 每个页面先展示路由权限。
- 勾选路由权限后，该页面查询权限默认生效。
- 功能权限只展示新增、删除、修改、操作，不单独展示查询。

功能权限：

| 权限 | 页面能力 |
| --- | --- |
| `action:role:create` | 创建角色 |
| `action:role:update` | 修改角色 |
| `action:role:delete` | 删除角色 |
| `action:role:operate` | 分配权限 |

约束：

- 内置角色不能删除。
- `admin` 角色默认拥有所有权限。
- 不支持角色继承。

### 11.7 退出登录

入口：顶部用户操作区。

行为：

- 调用退出接口。
- 清理前端 Token 和用户权限缓存。
- 跳转登录页。
- API 返回 401 时也执行本地退出流程。

## 12. 路由与菜单权限

路由权限用于控制导航显示和页面进入。

| 路由 | 页面 | 路由权限 | 默认查询权限 |
| --- | --- | --- | --- |
| `/` | 首页 | `route:dashboard` | 无 |
| `/users` | 用户管理 | `route:users` | `action:user:read` |
| `/permissions` | 权限管理 | `route:permissions` | `action:role:read` |
| `/profile` | 个人信息 | 登录即可访问 | 无 |

规则：

- 没有路由权限时，导航不显示。
- 没有路由权限时，即使手动输入地址也不能进入页面。
- 授予路由权限后，对应资源的查询权限默认生效。
- 页面内新增、删除、修改、操作按钮由功能权限控制。
- 实现时主框架页面、标题、菜单和 fallback 统一在 `web/src/router/page-registry.ts` 注册，新增页面流程见 `docs/development_page_guide.md`。
- 后端必须校验 API 权限，前端过滤不能作为安全边界。

## 13. 功能权限模型

功能权限固定为五类：

| 操作 | 编码后缀 | 含义 |
| --- | --- | --- |
| 增 | `create` | 新增资源 |
| 删 | `delete` | 删除资源 |
| 改 | `update` | 修改资源基础信息 |
| 查 | `read` | 查询列表、详情和字典 |
| 操作 | `operate` | 审核、启用、禁用、重置密码、分配角色、分配权限等业务动作 |

初版资源：

| 资源 | 权限 |
| --- | --- |
| 用户 | `action:user:create`、`action:user:read`、`action:user:update`、`action:user:delete`、`action:user:operate` |
| 角色 | `action:role:create`、`action:role:read`、`action:role:update`、`action:role:delete`、`action:role:operate` |

系统设置初版不做独立页面，可后续扩展为 `settings` 资源。

### 13.1 数据级权限预留规则

数据级权限暂不在初版实现，但后续业务模块需要预留接入边界。数据级权限只解决“同一个功能权限下，哪些数据可以看、哪些数据可以操作”的问题，不替代路由权限和功能权限。

权限分层：

| 层级 | 作用 | 示例 |
| --- | --- | --- |
| 路由权限 | 控制菜单和页面进入 | `route:tasks` |
| 功能权限 | 控制页面能力和 API 动作 | `action:task:download`、`action:task:stop` |
| 数据级权限 | 控制可访问的数据范围和行级操作 | 仅本人、仅本部门、仅分配给我的任务 |

后续可支持的数据范围：

| 范围 | 含义 |
| --- | --- |
| 全部 | 可查看或操作该资源的全部数据 |
| 公司 | 仅当前用户公司范围内的数据 |
| 部门 | 仅当前用户部门范围内的数据 |
| 本人 | 仅当前用户创建、负责或归属的数据 |
| 指派 | 仅明确指派给当前用户的数据 |
| 自定义 | 按业务资源扩展的条件，如任务类型、区域、数据源、项目等 |

后端预留规则：

- 数据级权限以后端为准，列表、详情、下载、启动、停止、删除等接口都必须在后端按当前用户和数据范围校验。
- 列表接口应在 repository 查询阶段完成数据范围过滤，不能先查出全部数据再依赖前端过滤。
- 详情和操作接口必须按资源 ID 再做一次数据范围校验，避免用户绕过列表直接请求接口。
- Service 方法应保留当前操作人上下文，后续接入数据级权限时可统一调用数据范围策略，不在页面或 API 层散落判断。
- 新业务模块不要硬编码角色名称判断数据范围，应通过角色、权限和后续数据范围策略共同决定。

前端预留规则：

- 页面入口和按钮显示先按功能权限控制，如 `authStore.has("action:task:download")`。
- 后续实现数据级权限后，列表接口应返回每行可执行动作，例如 `can_update`、`can_delete`、`can_download`、`can_start`、`can_stop` 或等价的 `allowed_actions`。
- 行级按钮最终应同时满足功能权限和后端返回的行级能力；没有行级能力时不显示或禁用对应按钮。
- 前端只做体验控制，不独立推导数据范围，不把公司、部门、本人等规则写死在页面组件里。
- 可复用按钮仍优先使用 `PermissionButton` 或统一权限辅助函数，后续再扩展为同时接收行级能力，避免每个页面重复写判断。

任务类页面建议拆分操作权限，不把所有业务动作都塞进 `operate`。例如：

| 权限 | 页面能力 |
| --- | --- |
| `route:tasks` | 进入任务页面 |
| `action:task:read` | 查看任务列表和详情 |
| `action:task:create` | 新增任务 |
| `action:task:update` | 修改任务配置 |
| `action:task:delete` | 删除任务 |
| `action:task:download` | 下载任务结果 |
| `action:task:start` | 启动任务 |
| `action:task:stop` | 停止任务 |

## 16. 初始化安装策略

系统首次启动时，如果未检测到安装配置或网站数据库未创建，Web 必须自动进入初始化安装页。初始化完成前，只开放健康检查、安装状态和安装提交接口；登录、注册、后台页面和受保护 API 均不可使用。

安装页内容：

- 数据库类型：SQLite 或 MySQL。
- SQLite 数据库文件路径，可留空使用默认运行目录。
- MySQL 连接信息：地址、端口、数据库名、用户名和密码。
- 初始管理员账号。
- 初始管理员密码。
- 初始管理员姓名、公司和部门。

安装行为：

- SQLite：自动创建数据库文件、基础表、内置权限、内置角色和初始管理员。
- MySQL：先连接 MySQL 服务端，自动创建指定数据库，再创建基础表、内置权限、内置角色和初始管理员。
- 安装完成后写入本地安装配置，只保存数据库连接配置和系统密钥，不保存管理员明文密码。
- 安装完成后重建数据库连接，前端跳转登录页。
- 已安装系统不允许重复执行初始化安装。

初始化内容：

- 创建基础表。
- 写入内置权限。
- 写入内置 `admin` 角色。
- 写入内置 `user` 角色。
- 给 `admin` 角色分配全部权限。
- 给 `user` 角色分配 `route:dashboard`。
- 创建初始管理员账号。

管理员约束：

- 初始管理员账号和密码由安装页填写，不在代码或文档中写死。
- 密码必须哈希保存。
- 不允许删除、禁用或移除最后一个管理员的管理员角色。
- 首次登录后建议管理员自行修改密码。

## 18. 安全约束

- 密码必须哈希保存。
- 不能明文保存密码。
- 登录失败返回统一错误。
- 禁用用户不能登录。
- 待审核和驳回用户不能登录。
- 修改密码必须校验旧密码。
- 忘记密码只能联系管理员重置。
- 不允许删除内置角色。
- 不允许删除或禁用最后一个管理员。
- 不允许普通管理员把自己降权到失去权限管理能力。
- 受保护 API 必须统一走鉴权和权限依赖。
- 前端权限控制只是体验优化，后端必须再次校验。
- 运行期不允许为了加载 UI、图标、字体或文档资源访问外网。

## 20. 实现顺序建议

后续进入代码实现时，建议按下面顺序：

1. 初始化 `server/` 和 `web/` 基础工程。
2. 建立前端页面壳、主题变量和离线资源策略。
3. 实现数据库配置、连接、会话和 repository 边界。
4. 实现 SQLite 初始化。
5. 实现注册、登录、退出、当前用户和鉴权依赖。
6. 实现权限校验和路由默认查询权限规则。
7. 实现首页。
8. 实现用户管理 API 和页面，并在用户管理中处理注册审核。
9. 实现个人信息 API 和页面。
10. 实现权限管理 API 和页面。
11. 做冗余代码清理、chunk 检查、离线资源扫描和断网验证。

确认这份设计后，再开始实现代码。
