# Metrix 平台总体设计

## 1. 定位与目标

Metrix 是一个面向数据接入、文件处理、数据库处理、脚本执行和任务调度的 Web 平台。平台不绑定单一报表或单一处理场景，容量报表、周期报表、临时分析脚本和后续其他数据处理需求都应作为业务模板或任务配置接入。

核心目标：

- 接入 FTP/SFTP、本地上传和后续其他数据源。
- 处理压缩包、CSV、XLSX 等常见数据文件。
- 将文件处理、数据入库、SQL 执行、脚本任务、结果查询统一到可追踪的任务链路中。
- 使用 Web 界面进行配置、运行、监控、查询和维护。
- 按职责拆分模块，每个核心模块独立 Git 仓库维护，主仓库通过 submodule 锁定版本。
- 后端使用 Python，前端只做 Web，不引入 Tauri 或桌面壳。

非目标：

- 第一版不做复杂分布式架构。
- 第一版不追求覆盖所有数据库，优先跑通 MySQL 主链路。
- 第一版不把所有业务规则做成插件系统，先以清晰配置和业务模板承载。
- 不把某个历史项目直接搬成一个巨型模块；只复用成熟思路和必要经验。

## 2. 技术选型

### 2.1 后端

建议使用 Python + FastAPI + Uvicorn。

选择理由：

- FastAPI 适合构建清晰的资源 API，自动生成 OpenAPI 文档，便于 Web 前端和外部系统调用。
- Python 适合文件处理、数据清洗、SQL 编排和脚本任务。
- Uvicorn 启动简单，适合 Windows 本地开发、内网服务器部署和容器部署。

配置与数据校验建议使用 Pydantic。第一版保持同步接口和同步数据库访问即可，只有出现明确并发瓶颈时再引入异步数据库驱动或任务队列。

### 2.2 前端

建议使用 Vue 3 + TypeScript + Vite + Naive UI。

选择理由：

- 和现有 Web 使用体验接近，开发成本低。
- TypeScript 能约束 API 响应、表单配置和任务状态。
- Vite 适合快速开发和 Web 构建。
- Naive UI 适合后台管理界面，表格、表单、弹窗、菜单、通知等组件比较完整。

前端只调用 API，不直接承载业务处理逻辑。

### 2.3 数据库

第一阶段以 MySQL 为主。数据库模块内部保留数据库适配边界，但不要为了未来多数据库提前堆复杂抽象。

建议策略：

- 平台元数据表和业务结果表先放在 MySQL。
- 数据库模块负责连接、库表、入库、SQL 执行、查询和导出。
- 文件处理模块不直接持有数据库连接。
- 后续确实需要 PostgreSQL、SQLite 或其他数据库时，再在数据库模块内增加适配器。

### 2.4 文件处理

CSV 处理优先采用流式或分块读取。XLSX 处理优先采用只读流式模式。Pandas 可以用于中小规模数据清洗和业务脚本，但不应成为所有入库链路的默认前置步骤。

建议原则：

- 大 CSV 不整文件载入内存。
- XLSX 按 Sheet 和行分批处理。
- 文件解析只输出标准化数据批次和元信息。
- 入库方式由数据库模块决定。

### 2.5 调度与任务

第一版建议使用进程内调度器承载固定周期扫描、手动任务和轻量定时任务。任务量上来后，再引入独立 worker 或队列。

调度能力应先满足：

- 手动上传处理。
- 远程目录扫描处理。
- 定时扫描和就绪判断。
- SQL 脚本任务。
- Python 脚本任务。
- 运行历史、日志、失败重试和任务取消。

## 3. 开发规范

### 3.1 基本原则

- 遵循 KISS 和 YAGNI，只解决当前真实需求。
- 每个模块只负责自己的职责，不跨边界处理其他模块细节。
- 优先使用已有能力和标准工具，不轻易引入新依赖。
- 配置、任务、脚本、文件、数据库操作都要可追踪。
- 临时文件、调试日志、测试产物和本地凭据不进入版本库。

### 3.2 Python 代码风格

- 包名和模块名使用 snake_case。
- 服务类和模型类职责要小，避免一个类同时做下载、解析、入库和脚本执行。
- API 层只负责请求校验、鉴权、调用服务和返回结果。
- 服务层负责业务编排，但不直接塞大量数据库或文件底层细节。
- 工具函数只放通用能力，不放业务流程。
- 异常要转换为统一错误结构，不能把底层异常原样暴露给前端。

### 3.3 前端代码风格

- 组件使用 PascalCase 命名。
- API 类型集中维护，页面组件不手写散落的响应结构。
- 页面负责交互状态，业务处理交给后端 API。
- 表单配置要有明确默认值、校验规则和保存反馈。
- 任务日志、进度、错误提示要能被用户快速定位，不用大量解释性文字堆页面。

### 3.4 文档规范

- 架构设计放在 `docs/design/`。
- 项目上下文和重要维护记录放在 `docs/project_context.md`。
- 模块仓库各自维护 README，但主仓库必须记录模块职责、版本和组合方式。
- 文档说明架构、边界、流程和约束，不写实现代码。

## 4. 主仓库与 Submodule 结构

Metrix 主仓库是平台组合仓库，负责锁定模块版本、维护平台级文档、部署说明、运行配置样例和集成脚本。

建议主仓库目录：

| 目录 | 说明 |
| --- | --- |
| `docs/design/` | 平台设计、模块边界、数据流和演进路线 |
| `docs/project_context.md` | 项目记忆和关键维护记录 |
| `web/` | 前端工程，前端相关 submodule 放在这里 |
| `server/` | 后端工程，后端相关 submodule 放在这里 |
| `deploy/` | 部署配置、容器配置、服务配置样例 |
| `configs/` | 平台配置样例，不放真实密码和私有地址 |
| `scripts/` | 平台级开发、构建、检查脚本 |
| `runtime/` | 本地运行目录，作为忽略目录处理 |

建议基础目录风格：

```text
Metrix/
  web/
    src/
      api/
      components/
  server/
    app/
      api/
```

Submodule 不再统一放到 `modules/`，而是按前后端归属放在 `web/` 或 `server/` 下。前端整体可以作为一个独立仓库挂到 `web/`；后端各能力模块放到 `server/app/` 的对应职责目录下。

建议 submodule 结构：

| 子目录 | 仓库名 | 职责 |
| --- | --- | --- |
| `web/` | `web-ui` | Web 前端界面 |
| `server/app/core` | `core` | 平台核心、认证、配置、状态、统一 API 壳 |
| `server/app/sources/remote` | `source-remote` | FTP/SFTP 数据源、远程扫描、下载、远端清理 |
| `server/app/files/processing` | `file-processing` | 解压缩、CSV/XLSX 读取、字段映射、数据标准化 |
| `server/app/database` | `database` | 数据库连接、库表管理、入库、SQL 执行、查询导出 |
| `server/app/jobs` | `job-runner` | 任务编排、定时调度、脚本执行、运行历史 |
| `server/app/templates` | `templates` | 业务模板、报表模板、脚本模板和配置模板 |
| `server/app/contracts` | `contracts` | API 契约、错误码、共享类型说明 |

第一阶段可以先保留核心的五个模块：`web-ui`、`core`、`source-remote`、`file-processing`、`database`。任务调度和模板复杂度上来后，再把 `job-runner`、`contracts`、`templates` 独立出来。

Submodule 管理原则：

- 主仓库只锁模块提交，不直接修改模块内部文件。
- 模块发生变更时，先在模块仓库提交，再更新主仓库 submodule 指针。
- 模块之间通过公开接口协作，不通过相对路径跨仓库读取内部文件。
- 主仓库记录每次模块升级的原因和影响。

## 5. 模块职责与边界

### 5.1 `core`

职责：

- 平台启动、认证、权限、Token、系统配置、审计日志。
- 统一错误码、任务状态枚举、基础模型和模块注册。
- 聚合各模块 API，形成 Web 平台后端入口。
- 管理平台运行目录、配置样例和健康检查。

边界：

- 不实现 FTP/SFTP 协议细节。
- 不解析 CSV/XLSX。
- 不直接执行 SQL 入库细节。
- 不承载具体报表业务规则。

### 5.2 `source-remote`

职责：

- FTP/SFTP 连接配置、连接测试、目录扫描。
- 远程文件列表、大小、修改时间、路径、基础日期信息。
- 按规则下载文件。
- 下载成功后的远端文件删除策略。
- 网络超时、重试、断连恢复和错误记录。

边界：

- 不解压文件。
- 不读取 CSV/XLSX 内容。
- 不写数据库。
- 不执行 SQL 或 Python 脚本。

### 5.3 `file-processing`

职责：

- 压缩包解压和原始文件识别。
- CSV 编码识别、分隔符识别、表头识别和分块读取。
- XLSX Sheet 过滤、表头识别、只读分批读取。
- 字段映射、类型转换、空值规范化、异常值处理。
- 文件名日期、目录周期、数据范围识别。
- 输出标准化数据批次和处理元信息。

边界：

- 不持有真实数据库连接。
- 不创建、删除、修改数据库表。
- 不执行业务 SQL。
- 不保存任务最终状态，只返回本阶段结果和错误。

### 5.4 `database`

职责：

- 数据库连接、连接池、连接测试和权限校验。
- 库管理、表管理、字段查看、索引查看。
- 表数据查询、分页、筛选、导出。
- 数据入库、临时表、正式表、批量加载和事务控制。
- SQL 脚本执行、执行日志、结果集返回。
- 平台元数据表初始化和迁移策略。

边界：

- 不扫描远程目录。
- 不解析压缩包和文件内容。
- 不决定任务什么时候运行。
- 不承载具体页面状态。

### 5.5 `job-runner`

职责：

- 手动任务、远程任务、定时任务和脚本任务编排。
- 任务状态、阶段、进度、日志、耗时、失败原因。
- 调度策略、就绪判断、任务互斥、取消、超时和重试。
- 调用远程模块、文件模块、数据库模块完成主链路。
- 管理运行历史、原始文件归档、结果资源和清理策略。

边界：

- 不实现文件解析细节。
- 不直接写数据库底层接口。
- 不在任务编排中写具体业务 SQL。
- 不让脚本绕过平台权限和审计。

### 5.6 `web-ui`

职责：

- 数据源配置、连接测试、远程目录浏览。
- 文件处理配置、字段映射、Sheet 过滤、日期规则。
- 数据库连接、库表管理、数据查询、SQL 执行。
- 任务中心、运行历史、日志查看、结果下载。
- 脚本中心、模板管理、API Token 和 API 文档。

边界：

- 不直接访问 FTP/SFTP。
- 不直接访问数据库。
- 不包含业务处理逻辑。
- 不通过本地文件路径绕开后端 API。

### 5.7 `contracts`

职责：

- 统一 API 响应结构。
- 错误码、状态码、任务阶段和常用枚举。
- 前后端共享的接口约定和版本说明。

边界：

- 不包含运行时业务逻辑。
- 不依赖具体数据库或文件处理实现。

### 5.8 `templates`

职责：

- 保存业务模板、字段映射模板、SQL 脚本模板、Python 脚本模板。
- 将不同报表和处理流程以配置化方式接入平台。
- 记录模板适用的数据源、目标表、脚本顺序和结果资源。

边界：

- 模板不能修改平台核心运行机制。
- 模板不能绕开数据库模块直接写库。
- 模板脚本必须经过任务模块执行并记录审计。

## 6. 文件处理与数据库入库边界

文件处理模块和数据库模块的职责必须严格分开。

文件处理模块负责回答：

- 原始文件是什么类型。
- 里面有哪些 Sheet、字段、行和数据范围。
- 字段如何映射到标准字段。
- 哪些值需要转换、清洗或标记异常。
- 可以向下游输出哪些数据批次。

数据库模块负责回答：

- 目标库是否可用。
- 目标表是否存在，字段是否匹配。
- 应该使用批量加载、分块插入还是临时表。
- 入库是否需要事务、去重、覆盖或追加。
- 入库后如何查询、导出和执行后续 SQL。

入库性能建议：

- 大 CSV 优先使用数据库原生批量加载能力。
- 原生批量加载不可用时，使用分块批量插入。
- XLSX 先分 Sheet、分块读取，再交给数据库模块入库。
- 复杂清洗只在必要时使用 Pandas，避免无意义整表加载。
- 每个入库任务必须记录文件名、行数、耗时、失败位置和目标表。

入库便利性建议：

- 常规报表通过字段映射和目标表配置完成。
- 临时分析允许创建一次性任务，但仍要保留任务历史。
- SQL 脚本可以接收任务参数，例如目标周期、批次 ID、目标表名。
- Python 脚本适合复杂处理，但脚本输入输出要通过平台约定管理。

## 7. 核心数据流

建议主链路如下：

| 阶段 | 执行模块 | 说明 |
| --- | --- | --- |
| 数据源扫描 | `source-remote` | 获取远程目录和文件清单 |
| 数据就绪判断 | `job-runner` | 根据周期、目录、文件名日期判断是否可处理 |
| 文件下载 | `source-remote` | 下载到原始数据目录 |
| 文件展开 | `file-processing` | 解压压缩包并识别 CSV/XLSX |
| 数据标准化 | `file-processing` | 完成字段映射、类型转换和数据批次输出 |
| 数据入库 | `database` | 按策略写入临时表或目标表 |
| 脚本执行 | `job-runner` + `database` | 执行 SQL 或 Python 脚本 |
| 结果查询 | `database` | 查询结果表、导出结果文件 |
| 历史归档 | `job-runner` | 记录原始数据、日志、结果和耗时 |
| 清理 | `job-runner` | 按策略清理临时文件和远端源文件 |

## 8. 核心模型

| 模型 | 说明 |
| --- | --- |
| 数据源 | FTP/SFTP、本地上传或未来扩展来源 |
| 文件清单 | 一次扫描得到的文件列表和基础元信息 |
| 数据集 | 一批准备处理的原始文件集合 |
| 处理配置 | Sheet 过滤、字段映射、日期规则、目标表规则 |
| 数据批次 | 文件处理模块输出的标准化数据单元 |
| 任务 | 一次手动、远程、定时或脚本处理动作 |
| 运行记录 | 任务的一次实际运行，包含阶段、状态、日志和耗时 |
| 脚本 | SQL、Python 或后续其他脚本类型 |
| 结果资源 | 结果表、导出文件、日志、原始数据归档 |
| 业务模板 | 某类报表或处理流程的配置组合 |

## 9. API 设计

API 应按资源分组，不按页面分组。

建议分组：

- 认证与 Token API。
- 数据源 API。
- 文件清单与数据集 API。
- 文件处理配置 API。
- 数据库连接、库表、数据查询 API。
- SQL 与脚本 API。
- 任务与调度 API。
- 历史与结果资源 API。
- 业务模板 API。
- 系统配置与健康检查 API。

API 约定：

- 使用统一响应结构。
- 错误码可被前端识别和展示。
- 任务型接口返回任务 ID，由任务状态接口轮询或订阅。
- 查询型接口必须分页。
- 导出型接口要记录历史并按策略清理临时文件。
- API 文档由后端自动生成，并提供中文说明。

## 10. 调度设计

调度模块负责判断“什么时候运行”和“运行到哪一步”，不负责文件解析和数据库底层实现。

建议能力：

- 固定间隔扫描远程目录。
- 按文件名日期或业务日期判断数据是否齐全。
- 支持日粒度、周粒度和自定义周期。
- 支持多个目录共同就绪后触发。
- 可配置空目录是否视为停推并跳过。
- 就绪后先记录就绪状态，下一轮再触发下载处理，降低半成品文件风险。
- 处理成功后再清理远端源文件。
- 失败时保留就绪状态和错误记录，便于重试。

调度状态需要在 Web 页面可见，包括最近扫描时间、目标周期、目录状态、失败次数、下一次扫描时间和当前任务 ID。

## 11. 脚本设计

脚本是平台的重要能力，但不能绕开任务、权限和审计。

SQL 脚本：

- 由数据库模块执行。
- 可以读取任务参数和模板参数。
- 执行前应记录脚本版本或内容摘要。
- 执行后记录影响行数、耗时和错误。

Python 脚本：

- 由任务模块统一启动。
- 输入来自平台约定的参数、数据批次、表名或文件路径。
- 输出结果必须登记为结果资源或写入数据库模块管理的目标表。
- 需要限制执行目录、环境变量、超时和可访问资源。

脚本中心应支持保存、查看、执行、复制、禁用和历史追踪。第一版可以先做简单版本记录，后续再增强审批和发布流程。

## 12. 存储与历史

建议运行目录分层：

| 目录概念 | 说明 |
| --- | --- |
| 原始数据目录 | 保存上传或远程下载的原始文件 |
| 临时处理目录 | 保存解压、转换和入库中间文件 |
| 归档目录 | 保存可追溯的任务原始数据和日志 |
| 导出目录 | 保存查询导出的临时文件 |
| 脚本目录 | 保存 SQL、Python 和模板脚本 |
| 日志目录 | 保存平台运行日志和任务日志 |

历史能力：

- 按任务查看阶段、日志、耗时和结果。
- 支持下载原始数据包。
- 支持下载单个结果文件。
- 支持按天数、次数或空间占用清理。
- 临时目录可以清理，归档目录必须按配置清理，不能误删仍需追溯的数据。

## 13. 配置与安全

配置分层：

- 平台配置：端口、运行目录、日志、历史保留。
- 认证配置：用户、Token、权限策略。
- 数据源配置：协议、地址、端口、账号、目录、超时、删除源文件策略。
- 文件处理配置：文件类型、Sheet 过滤、字段映射、日期解析、异常值策略。
- 数据库配置：连接、目标库、入库策略、目标表规则。
- 任务配置：调度周期、并发限制、失败重试、超时。
- 脚本配置：脚本类型、执行参数、权限、超时。

安全原则：

- 真实账号、密码、Token、私有地址不提交版本库。
- Token 只展示一次或按权限展示。
- SQL 执行和 Python 脚本执行必须有权限控制。
- 删除远端源文件必须有明确开关，并只删除已成功处理的文件。
- 所有高风险操作记录审计日志。

## 15. 部署与运行

第一版建议支持：

- Windows 本地开发。
- 内网服务器 Web 部署。
- Docker 部署。

部署形态：

- Web 前端构建后由后端或独立 Web 服务托管。

## 16. 版本与协作

Git 规则：

- 主仓库维护 submodule 指针、平台文档和部署说明。
- 子模块仓库维护各自代码、测试和 README。
- 子模块变更先在子模块提交，再在主仓库提交 submodule 指针更新。
- 提交信息使用统一格式，描述真实变更。

分支建议：

- 主线分支保持可运行。
- 功能分支按模块建立。
- 跨模块变更要在主仓库记录影响范围。

## 17. 开发路线

### 阶段一：平台骨架

- 建立主仓库和核心 submodule。
- 完成后端入口、前端壳、认证、健康检查。
- 完成数据源配置和 FTP/SFTP 连接测试。
- 完成基础项目文档和开发规范。

### 阶段二：文件到数据库主链路

- 支持本地上传和远程下载。
- 支持 ZIP、CSV、XLSX 基础处理。
- 支持字段映射和日期规则。
- 支持 MySQL 入库、表管理和数据查询。
- 建立任务状态、日志和历史记录。

### 阶段三：脚本与调度

- 支持 SQL 脚本执行。
- 支持 Python 脚本执行。
- 支持定时扫描、就绪判断和失败重试。
- 支持处理成功后清理远端源文件。

### 阶段四：业务模板化

- 将不同报表处理规则沉淀为模板。
- 支持模板参数、脚本顺序、目标表和结果资源配置。
- 支持从模板快速创建任务。

### 阶段五：平台增强

- 增加更细的权限和审计。
- 增强任务并发、队列和 worker。
- 增加更多数据库适配。
- 增加告警、监控和模块版本管理。

## 18. 关键风险与决策

| 风险 | 处理建议 |
| --- | --- |
| 模块拆得太细导致开发慢 | 第一版只拆真实核心边界，复杂后再独立更多模块 |
| 文件模块和数据库模块职责混乱 | 文件模块只输出标准化数据批次，数据库模块只负责落库和查询 |
| 大文件性能不足 | CSV 分块、XLSX 只读流式、MySQL 原生批量加载优先 |
| 调度误判半成品文件 | 使用就绪状态和下一轮触发机制，避免推送刚完成就处理 |
| 脚本权限风险 | 脚本统一由任务模块执行，记录权限、参数、版本和审计 |
| submodule 管理复杂 | 主仓库锁版本，模块仓库各自提交，跨模块变更写清影响 |
| 业务规则污染核心 | 报表规则进入模板或业务模块，不进入核心模块 |

## 19. 近期落地建议

近期不要急着做完整平台。建议先建立 Metrix 主仓库和最小模块组合，跑通一条稳定链路：

1. 配置一个 FTP/SFTP 数据源。
2. 扫描并下载一批文件。
3. 解压并读取 CSV/XLSX。
4. 按字段映射写入 MySQL。
5. 执行一段 SQL 脚本。
6. 在 Web 上查看任务状态、日志、结果表和导出文件。

这条链路跑稳之后，再逐步增加业务模板、复杂调度、Python 脚本和更多数据库能力。
